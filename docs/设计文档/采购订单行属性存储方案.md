# 采购订单行属性存储方案

> 日期：2026-07-16
> 状态：已决策

## 一、问题背景

采购订单行（`qxx_pur_order_line`）原设计仅针对纸张采购场景，硬编码了 4 个纸张专属字段：`paper_width`（门幅）、`paper_weight`（克重）、`paper_type`（种类）、`roll_count`（卷数）。

实际业务中采购对象不限于纸张，还可能是：
- **成品**（产品尺寸、装箱规格、印刷要求）
- **辅料**（胶水/油墨/绳子等，属性各异）
- **外协加工**（印刷、工艺加工等，加工要求描述）
- **纸袋成品**（绳料规格、口部提拔、底板类型）

每种采购对象有不同属性集，且未来可能新增类型。固定平铺纸张字段无法覆盖这些场景。

## 二、方案对比

### 方案 A：平铺加列

在 `qxx_pur_order_line` 表上持续 ALTER TABLE 加列。

| 优点 | 缺点 |
|------|------|
| 查询最简单（单表） | 列数无上限增长，10 种类型 = 30-50 列 |
| 与现有销售行/工单一致 | 新增属性类型必须 DDL 迁移 |
| | 表结构频繁变更 |

**否决理由**：列数随行业类型线性膨胀，长期维护成本高。

### 方案 B：行业子表

每种属性类型建一张子表（类似物料主数据的 `qxx_md_item_attr_paper` 模式）。

| 优点 | 缺点 |
|------|------|
| 结构清晰，类型化 | 新增类型仍需建表 + Entity + Mapper + Service |
| 按需 JOIN | 子表 factory_id 与主表冗余，有一致性风险 |
| | 全项目要建大量子表（采购行/销售行/工单 × N 种类型） |

**否决理由**：只是把"加列"变成"加表"，本质一样。子表数量 = 单据类型 × 属性类型，过多。

### 方案 C：JSON 列存储 ✅ 已选

采购行表新增一个 `line_attrs JSON` 列，所有行业属性存 JSON。

| 优点 | 缺点 |
|------|------|
| 新增属性类型零 DDL | 无法 DB 级别列约束（属性本就可选，可接受） |
| 表结构永不变 | 查询过滤需 JSON 函数（采购行按 order_id 查，极少按属性过滤） |
| factory_id 只在主表，无子表冗余 | 需 JSON TypeHandler（标准模式，不复杂） |
| 为将来低代码留路 | |
| MySQL 8.0 原生支持 JSON | |

### 方案 D：低代码 / 元数据驱动

元数据表定义属性模板 + 属性字段，前端动态渲染，后端存 JSON。

| 优点 | 缺点 |
|------|------|
| 真正零代码扩展 | 开发量大（元数据表 + 模板管理 UI + 动态渲染引擎） |
| 非开发人员可配置 | 当前属性类型仅 3-4 种，过度设计 |
| | 调试困难 |

**否决理由**：当前属性类型少且稳定，低代码投入产出比低。但 JSON 存储（方案 C）是低代码的前置条件——将来需要时加元数据层即可，存储层不用动。

## 三、多工厂架构分析

本系统每个表都有 `factory_id`，支持多工厂共用一套系统。不同工厂采购不同类型物料：

| 工厂类型 | 关心属性 | 不关心属性 |
|---------|---------|-----------|
| 纸袋厂 | 纸张（门幅/克重/种类/来源） | 产品尺寸 |
| 礼品盒厂 | 成品（产品尺寸/印刷要求） | 纸张克重 |
| 未来新工厂 | 可能有全新属性 | - |

**JSON 方案在多工厂下的优势**：
- factory_id 只在主表做隔离，无子表 factory_id 冗余问题
- 不同工厂在 `line_attrs` JSON 中填不同的 key，互不干扰
- 新工厂类型不需要任何 Schema 变更

## 四、架构决策

**采用方案 C：JSON 列存储。**

核心理由：
1. **交易单据用 JSON/平铺，主数据用子表** — 物料是长期主数据（可能新增属性类型）用子表；采购行是交易快照用 JSON
2. **多工厂友好** — factory_id 只在主表，无子表一致性风险
3. **扩展性最优** — 新增属性类型零 DDL，只加 JSON key
4. **为低代码留路** — 将来加元数据层即可，存储层不用动
5. **投入产出比最优** — 一个 JSON 列 + TypeHandler，比建 N 张子表简单得多

## 五、JSON 结构规范

```json
{
  "paper": {
    "width": "925",
    "weight": "120",
    "source": "联盛A2",
    "type": "箱板纸",
    "rollCount": 5
  },
  "paperBag": {
    "ropeSpec": "7.5cm黄牛皮圆纸绳",
    "mouthType": "锯齿口",
    "bottomType": "灰底白板"
  },
  "product": {
    "size": "254*127*330mm",
    "packageSpec": "250个/箱",
    "printingReq": "1色满版黑印刷"
  }
}
```

命名规则：
- 一级 key 为属性分组名（paper/paperBag/product），与物料子表对应
- 二级 key 为具体属性名，camelCase
- 值类型：字符串为主，rollCount 为数字

## 六、信息流设计

```
选择物料 -> getItem(itemId) 获取物料完整属性（含子表）
  -> 映射为 lineAttrs JSON 存入采购行（快照）
  -> 用户可覆盖属性值
  -> 保存到采购行 line_attrs 列
  -> 不回写物料主数据
```

**属性覆盖的流程引导**：
- 采购行属性是"本次采购的临时要求"快照，不回写物料主数据
- 后续环节（入库/库存/生产）默认看物料标准属性
- 如需后续环节感知差异（长期变更规格），用户应前往物料管理新建物料变体
- 采购行表单属性 Tab 顶部提示此规则

## 七、迁移策略

现有 4 个纸张平铺列（paper_width/paper_weight/paper_type/roll_count）：
1. Flyway 新增 `line_attrs JSON` 列
2. 将现有纸张数据迁移进 JSON 的 `paper` 对象
3. 删除旧平铺列

## 八、后续扩展

采购行先行采用 JSON 方案。销售订单行、工单的平铺属性字段后续可按同样模式改造为 `line_attrs JSON`，统一架构。

将来如需低代码能力，在 JSON 存储之上加：
- `qxx_md_attr_template`（属性模板定义表）
- `qxx_md_attr_field`（属性字段定义表）
- 前端动态渲染引擎
- 存储层（JSON 列）不用变
