# 领料单完整生命周期设计

> 版本：v2.0（2026-07-07 完整生命周期改造）
> 适用模块：`backend/.../mes/wm/issue`、`frontend/src/views/mes/wm/issue`、`app/pages/mes/wm/issue`
> 关联迁移：V43（建表+字段增强）、V44（字典+权限）、V45（编码规则）、V46（issue_line 补列）

## 1. 设计目标

领料单是生产执行阶段的核心单据，承载"工单需求 → 仓库发料 → 产线消耗"的物流闭环。本设计实现：

- **完整状态机**：从制单到关闭的 8 态流转，支持审核闸口、分批发料、作废回退
- **库存两阶段扣减**：预占（available）+ 实扣（onhand）分离，避免确认即扣导致的退料复杂
- **并发安全**：Redisson 分布式锁 + DB 行锁 + 事务模板三重保障
- **全链路追溯**：库存流水（transaction）+ 物料追溯（material_trace）双向可审计
- **多端协同**：PC 工作台（计划/仓管）+ 移动 PDA（车间扫码发料）

## 2. 状态机（标准制造 8 态）

```
        提交审核            审核通过              预占库存
DRAFT ─────────▶ PENDING ─────────▶ APPROVED ─────────▶ ALLOCATED
 │  (制单)        │  (待审)          │  (已下达)          │  (已预占)
 │                │ 审核退回          │                   │
 │                ◀──────────────────│                   │ 发料(可分批)
 │                │                   │                   ▼
 │                │                   │            PARTIAL_ISSUED ──继续发料──▶ (循环)
 │                │                   │                   │ 全部发完
 │                │                   │                   ▼
 │                │                   │                ISSUED ──收料确认/关闭──▶ CLOSED
 │                │                   │
 └──────────作废─────────────────────┘
                    │
                    ▼
                CANCELED (终态)
```

### 2.1 状态定义（字典 `mes_wm_issue_status`）

| 状态值 | 名称 | 含义 | 库存影响 | 可流转到 |
|--------|------|------|---------|---------|
| `DRAFT` | 草稿 | 制单中，可编辑删除 | 无 | PENDING、CANCELED |
| `PENDING` | 待审核 | 已提交，等计划员/仓管主管审核 | 无 | APPROVED、DRAFT（退回）、CANCELED |
| `APPROVED` | 已下达 | 审核通过，仓库可见，未预占 | 无 | ALLOCATED、CANCELED |
| `ALLOCATED` | 已预占 | 已扣减 `quantity_available` | available ↓ | APPROVED（释放）、PARTIAL_ISSUED、ISSUED、CANCELED |
| `PARTIAL_ISSUED` | 部分发料 | 分批发料中，onhand 部分扣减 | onhand 部分 ↓ | ISSUED、CLOSED、（不可作废，已扣库存） |
| `ISSUED` | 已发料 | 全量出库完成，onhand 全扣 | onhand ↓ | CLOSED、（不可作废） |
| `CLOSED` | 已关闭 | 产线收料确认/手工关闭（终态） | 无 | 终态 |
| `CANCELED` | 已作废 | 作废（终态），ALLOCATED 态作废先恢复 available | available ↑（恢复） | 终态 |

### 2.2 状态流转动作

| 动作 | 接口 | 方法 | 源状态 → 目标状态 | 权限 | 说明 |
|------|------|------|------------------|------|------|
| 提交审核 | `PUT /submit/{id}` | `submitForApprove` | DRAFT → PENDING | `mes:wm:issue:submit` | 校验必须有明细行 |
| 审核通过 | `PUT /approve/{id}` | `approve` | PENDING → APPROVED | `mes:wm:issue:approve` | 记录 approveBy/approveTime |
| 审核退回 | `PUT /reject/{id}` | `reject` | PENDING → DRAFT | `mes:wm:issue:approve` | |
| 预占 | `PUT /confirm/{id}` | `confirmIssue` | DRAFT/APPROVED → ALLOCATED | `mes:wm:issue:edit` | 扣 available，写 ALLOCATE 流水 |
| 释放预占 | `PUT /release/{id}` | `releaseAllocation` | ALLOCATED → APPROVED | `mes:wm:issue:edit` | 恢复 available，写 RELEASE 流水 |
| 发料出库 | `PUT /issueOut/{id}` | `issueOut` | ALLOCATED/PARTIAL_ISSUED → PARTIAL_ISSUED/ISSUED | `mes:wm:issue:issueOut` | 分批扣 onhand，写 detail+ISSUE_OUT 流水+trace |
| 关闭 | `PUT /close/{id}` | `close` | ISSUED/PARTIAL_ISSUED → CLOSED | `mes:wm:issue:close` | 终态 |
| 作废 | `PUT /cancel/{id}` | `cancel` | 非终态 → CANCELED | `mes:wm:issue:cancel` | ALLOCATED 先恢复库存；ISSUED 拒绝作废 |
| 删除 | `DELETE /{ids}` | `delete` | 仅 DRAFT/PENDING | `mes:wm:issue:remove` | 级联删行 |

### 2.3 向后兼容

- `confirmIssue`/`executeIssue` 旧接口保留（@Deprecated），工单齐套看板等历史调用方不受影响
- 所有状态校验兼容历史值 `CONFIRMED`（视为 ALLOCATED）、`POSTED`（视为 ISSUED）
- 历史数据迁移：无需脚本，老数据状态值在运行时被识别为等价新状态

## 3. 数据模型

### 3.1 三层主从结构

```
WmIssueHeader（领料单头 qxx_wm_issue_header）
   ├─ WmIssueLine（领料行 qxx_wm_issue_line）        一对多，按 itemId 汇总
   │    └─ WmIssueDetail（发料明细 qxx_wm_issue_detail） 一对多，按批次/库位拆分的实际发料记录
   └─ WmTransaction（库存流水 qxx_wm_transaction）     每次 ALLOCATE/RELEASE/ISSUE_OUT 写入
```

- **Header**：单据头，记录状态、工单关联、仓库、申请总量、已发料累计、审核信息、乐观锁版本
- **Line**：领料行，每行一个物料的申请量（quantityIssue）和已发料累计（quantityIssued）
- **Detail**：发料明细，每次分批发料记录（按批次/库位拆分），关联 materialStockId

### 3.2 关键字段（Header，V43 增强后）

| 字段 | 类型 | 说明 |
|------|------|------|
| `issue_code` | varchar(64) | 唯一编码（uk_issue_code），自动生成 ISS+yyyyMMdd+3位 |
| `status` | varchar(50) | 8 态状态机 |
| `total_quantity` | decimal(14,4) | 申请总量 |
| `quantity_issued_total` | decimal(14,4) | 已发料累计（分批累加） |
| `quantity_received` | decimal(14,4) | 已收料累计（产线确认） |
| `approve_by` / `approve_time` | varchar/datetime | 审核人/时间 |
| `cancel_reason` | varchar(500) | 作废原因 |
| `version` | int | 乐观锁版本号 |

### 3.3 库存定位键（物料批次定位）

发料/预占时按以下复合键定位库存记录（`qxx_wm_material_stock`）：

```
itemId + batchId + warehouseId + vendorId(0) + workorderId(0) + qualityStatus("NORMAL")
```

> 说明：vendorId/workorderId 用 `0` 非 NULL（MySQL 唯一索引不认 NULL）。当前默认只发"正常"质量状态库存，后续可扩展按供应商/工单/质量分批出库。

## 4. 库存联动机制

### 4.1 两阶段扣减

| 阶段 | 时机 | 库存字段 | 事务类型 | 数量符号 |
|------|------|---------|---------|---------|
| 预占 | confirm（APPROVED→ALLOCATED） | `quantity_available` ↓ | `ALLOCATE` | 负数 |
| 释放预占 | release（ALLOCATED→APPROVED） | `quantity_available` ↑ | `RELEASE` | 正数 |
| 实扣 | issueOut（ALLOCATED→ISSUED） | `quantity_onhand` ↓ | `ISSUE_OUT` | 负数 |
| 作废恢复 | cancel（ALLOCATED→CANCELED） | `quantity_available` ↑ | `RELEASE`（内部调 release） | 正数 |
| 退料入库 | RtIssue.executeReturn | `quantity_onhand` ↑ | `RETURN_IN` | 正数 |

**设计要点**：预占只扣 available 不动 onhand；实扣只扣 onhand 不覆盖 available（executeIssue 显式 `setQuantityAvailable(null)` 防动态 UPDATE 误写）。这样并发 confirm 与 execute 互不干扰。

### 4.2 并发安全（三重保障）

```java
// WmIssueHeaderServiceImpl.confirmIssue
lockTemplate.execute("wm:issue:lock:" + issueId, 10,        // ① Redisson 分布式锁（单据级）
    () -> txTemplate.execute(status -> doConfirmIssue(issueId)));  // ② 事务模板（30s 超时）
// ③ doConfirmIssue 内：loadMaterialStockForUpdate（SELECT ... FOR UPDATE 行锁）
```

- **外层**：Redisson 分布式锁，粒度到单据 `wm:issue:lock:{issueId}`，等待 10s
- **中层**：TransactionTemplate 手动事务（30s 超时），禁止 `@Transactional`（库存变更必须先锁后事务）
- **内层**：DB 行锁 `SELECT ... FOR UPDATE`，防并发超卖

### 4.3 分批发料（issueOut）

```
输入：issueId + List<WmIssueDetail>（本次发料明细，含 lineId/itemId/batchCode/quantity）
处理：
  for each detail:
    1. FOR UPDATE 锁库存行
    2. 校验 onhand - quantity >= 0
    3. onhand -= quantity（不动 available，已在 confirm 扣过）
    4. 写 WmIssueDetail（materialStockId 关联）
    5. 写 WmTransaction（ISSUE_OUT，负数）
    6. 写 ProMaterialTrace（traceType=ISSUE）
    7. 累加 WmIssueLine.quantityIssued
  累加 header.quantityIssuedTotal，判断：
    - 全发完（累计 >= 申请总量）→ ISSUED
    - 否则 → PARTIAL_ISSUED
```

## 5. 与其他模块的联动

### 5.1 生产工单（强联动）

- **齐套分析生成领料单**：`ProWorkorderDocServiceImpl.generateIssueDocuments` 按 BOM×工序分组生成领料单
- **取消工单自动作废**：`ProWorkorderServiceImpl.cancelWorkorder` 取消工单时，自动作废工单下所有非终态领料单（ALLOCATED 态自动恢复库存）
- **齐套看板状态汇总**：`ProWorkorderDocServiceImpl` 聚合 draft/confirmed/posted 计数（兼容新 8 态映射）

### 5.2 退料（wm/rt-issue）

- 退料单 `WmRtIssue` 可从已发料领料单生成
- 退料执行 `executeReturn` 增加 onhand + 写 RETURN_IN 流水 + 写 trace（traceType=RETURN）
- 工单完成后按"已发料 - 已报工消耗 = 应退"自动生成退料建议

### 5.3 物料追溯（pro/material-trace）

- 发料写 `ProMaterialTrace(traceType='ISSUE', parentType='MATERIAL_STOCK', childType='CARD')`
- 实现"库存 → 流转卡"正向追溯
