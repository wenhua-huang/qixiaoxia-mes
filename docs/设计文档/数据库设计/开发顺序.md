# MES 数据库模块开发顺序

> 以**依赖最小**原则排列。被依赖的先做，依赖最多的后做。
> 核心思路：把 wm 拆为 wm-core（核心库存）和 wm-business（业务集成），让库存先行。

## 开发顺序

### ✅ 已完成

| 批次 | 模块 | 表数 | 依赖 |
|------|------|------|------|
| — | `mes-md.sql` | 17 | RuoYi `sys_user`、`mes-dv`(仅 `workstation_machine` 引用 `qxx_dv_machinery`) |
| — | `mes-dv.sql` | 12 | `qxx_md_factory`、`qxx_md_workshop` |

### 第 1 批：系统使能 + 核心库存（仅依赖 md+sys_user）

| 顺序 | 模块 | 表数 | 依赖 | 说明 |
|------|------|------|------|------|
| **1** | `mes_sys.sql` | 6 | `qxx_md_factory`、`sys_user` | 自动编码规则、附件、消息、待办。**所有模块都依赖它**。 |
| **2** | `mes-wm-core.sql` | 28 | `qxx_md_*` + `mes_sys` | 仓库结构、实时库存(`material_stock`)、库存变动(`transaction`)、物料入库、杂项出入、调拨、盘点、退货、条码。**pro/qc/pur 都需要库存表**。`material_stock.workorder_id` 默认 0（不适用），待 pro 开发后自然过渡。 |

> **wm-core 拆出后，`material_stock`、`transaction`、`batch` 在 pro/qc 之前就绪，后续模块可直接引用。**

### 第 2 批：独立低依赖模块（仅依赖 md+sys，可能引用 wm-core）

| 顺序 | 模块 | 表数 | 依赖 | 说明 |
|------|------|------|------|------|
| **3** | `mes-cal.sql` | 7 | `qxx_md_factory` | 班组、班次、排班、节假日。pro 排产用。 |
| **4** | `mes-tm.sql` | 2 | `qxx_md_factory` | 工装夹具类型+台账。完全独立，体量最小。 |
| **5** | `mes-pur.sql` | 2 | `qxx_md_factory/vendor/item` + `qxx_wm_arrival_notice`(可后补) | 采购订单。`arrival_notice_id` FK 可后加。 |

### 第 3 批：生产 + 质量（依赖 md+dv+sys+wm-core）

| 顺序 | 模块 | 表数 | 依赖 | 说明 |
|------|------|------|------|------|
| **6** | `mes-pro.sql` | 21 | `qxx_md_*` + `qxx_dv_machinery` + `qxx_wm_*`(库存相关) | 工单、工艺路线、任务、报工、流转卡、追溯。引用的 wm 表（`issue_header/detail`、`transaction`）在 wm-business 之前，FK 可先指向 wm-core 已有表或后补。 |
| **7** | `mes-qc.sql` | 17 | `qxx_md_*` + `qxx_pro_workorder/task/process` + `qxx_wm_batch` + wm 通知单 | IQC/IPQC/OQC/RQC。wm 通知单 FK 可后补。 |

### 第 4 批：业务集成收尾（依赖全部前置模块）

| 顺序 | 模块 | 表数 | 依赖 | 说明 |
|------|------|------|------|------|
| **8** | `mes-wm-business.sql` | 23 | pro + qc + pur | 生产领料/退料、产品产出/入库、外协领料/入库、备料通知。此时 pro/qc/pur 全部就绪，可一次性正确定义所有跨模块 FK。 |

## wm 拆分明细

### wm-core（28 表，仅依赖 md，第 1 批）

| 功能域 | 表名 |
|--------|------|
| 仓库结构 | `qxx_wm_warehouse`、`qxx_wm_storage_location`、`qxx_wm_storage_area` |
| **实时库存** | **`qxx_wm_material_stock`** |
| **库存变动** | **`qxx_wm_transaction`** |
| 批次管理 | `qxx_wm_batch` |
| 物料入库 | `qxx_wm_item_recpt`、`_line`、`_detail` |
| 杂项入库 | `qxx_wm_misc_recpt`、`_line` |
| 杂项出库 | `qxx_wm_misc_issue`、`_line` |
| 调拨转移 | `qxx_wm_transfer`、`_line`、`_detail` |
| 盘点 | `qxx_wm_stock_taking_plan`、`qxx_wm_stock_taking` |
| 供应商退货 | `qxx_wm_rt_vendor`、`_line`、`_detail` |
| 销售出库 | `qxx_wm_product_sales`、`_line`、`_detail` |
| 销售退货 | `qxx_wm_rt_sales`、`_line`、`_detail` |
| 条码 | `qxx_wm_barcode` |
| 装箱 | `qxx_wm_package`、`_line` |
| 纸卷明细 | `qxx_wm_roll_detail` |

> `qxx_wm_material_stock.workorder_id` 定义 `DEFAULT 0 NOT NULL`，0 = 不适用。pro 开发后无需改表即可按工单隔离库存。
> 物料入库单的 `pur_order_id`、`iqc_id` 均可为空，入库单可独立运作。

### wm-business（23 表，依赖 pro/qc/pur，第 4 批）

| 功能域 | 表名 | 核心依赖 |
|--------|------|----------|
| 生产领料 | `qxx_wm_issue_header`、`_line`、`_detail` | `qxx_pro_workorder`、`qxx_pro_task` |
| 生产退料 | `qxx_wm_rt_issue`、`_line`、`_detail` | `qxx_pro_workorder`、`qxx_qc_rqc` |
| 产品产出 | `qxx_wm_product_produce`、`_line`、`_detail` | `qxx_pro_workorder`(NOT NULL)、task/process/feedback |
| 产品入库 | `qxx_wm_product_recpt`、`_line`、`_detail` | `qxx_pro_workorder`、`qxx_qc_ipqc`、product_produce |
| 外协领料 | `qxx_wm_outsource_issue`、`_line`、`_detail` | `qxx_pro_workorder` |
| 外协入库 | `qxx_wm_outsource_recpt`、`_line`、`_detail` | `qxx_pro_workorder`、`qxx_qc_ipqc`、outsource_issue |
| 备料通知 | `qxx_wm_materialrequest_notice`、`_line` | `qxx_pro_workorder`(NOT NULL) |

## 关键决策记录

1. **wm 拆分为 core + business**：core 仅依赖 md，库存先行就绪供 pro/qc 引用；business 收尾时所有前置模块已就绪，一次性定义完整 FK。
2. **`material_stock.workorder_id` 归入 core**：虽语义关联 pro，但 `DEFAULT 0` 使表可独立运行；pro 开发后自然过渡为按工单隔离，无需 DDL 变更。
3. **通知单表（arrival_notice、sales_notice）暂未在 wm 中建表**，qc/pur 对其的引用 FK 可后补。
4. **外键可选的一律按可选处理**：如 `item_recpt.pur_order_id`、`product_sales.oqc_id`，不阻塞核心功能先行。
