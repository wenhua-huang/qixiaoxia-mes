# 数据库设计决策

> 与客户沟通确认后的关键设计决策。新增 MES 功能时必须遵循这些约定。

## 库存维度分层（成品变体 / 原料批次）

**原则**：`qxx_wm_material_stock` 一张表管全部库存，但不同物料身份的库存颗粒度不同。

| 物料身份 | 库存主维度 | 批次/卷号 | 说明 |
|---------|-----------|----------|------|
| 成品 FINISHED | `item_id`（变体=独立 item） | `batch_code` / `workorder_id` | 每个变体是独立的 `item_id`（通过 `parent_id` 指向父 SPU） |
| 半成品 SEMI | `item_id` | `batch_code` | 同上，按需建变体 |
| 原料 RAW | `item_id` | `lot_number` / `paper_roll_no` | — |
| 辅料 AUXILIARY | `item_id` | `batch_id` | — |
| 包材 PACK | `item_id` | `batch_id` | — |

**Why**: 成品的不同变体（彩印红绳 ≠ 黑印黄绳）是**不可互换的商品**，必须作为独立 `item_id` 管理库存。原料的不同批次是**可互换的物料**（白牛皮差 1mm 门幅不耽误用）。

### 库存唯一性约束

**双重保障**：A. 应用层 UPSERT（INSERT 前查重 → 查到则 UPDATE 数量） + B. 数据库唯一索引兜底。

**唯一键 = `factory_id + item_id + batch_id + warehouse_id + vendor_id + workorder_id + quality_status`**

`batch_id`/`vendor_id`/`workorder_id` 用 `0` 而非 `NULL`，因为 MySQL 唯一索引不认 NULL。

```
原料（纸张）= f1 + i100 + b5  + wh10 + v3 + wo0  + NORMAL  ← vendor=德欣纸业
成品（纸袋）= f1 + i200 + b8  + wh20 + v0 + wo15 + NORMAL  ← workorder=WO-001
辅料（胶水）= f1 + i300 + b0  + wh30 + v0 + wo0  + NORMAL  ← 不管批次/供应商/工单
```

## SPU/SKU 实现

**`qxx_md_item.parent_id` 自引用**（参考 Qcadoo MES 的 `basic_product.parent_id` 设计）。

```
qxx_md_item:
  item_id=201  奔趣纸袋 PICO-001      parent_id=0    ← SPU（父产品，定义标准规格）
  item_id=202  奔趣纸袋-彩印红绳       parent_id=201  ← 变体（覆盖印刷/绳料/包装字段）
  item_id=203  奔趣纸袋-黑印黄绳       parent_id=201  ← 变体
  item_id=204  奔趣纸袋-现货标准款     parent_id=201  ← 变体（全部字段继承，等于SPU本身）
```

变体 `item_id` 上差异字段（`printing_req`/`rope_spec`/`package_spec`）有值 → 覆盖父产品；为 null → 继承父产品。**不需要独立的 SKU 表**，库存、BOM、工单全部走 `item_id`，零额外逻辑。

**写入时机**：物料建档时可自动建标准变体（`parent_id=父ID，所有差异字段=null`）；首次接客户定制时业务员在 `qxx_md_item` 界面新建变体（填差异字段）；返单时选已有变体。

## 表结构设计约定

| 决策 | 结论 | 原因 |
|------|------|------|
| 纸张行业扩展字段（门幅/克重/印刷要求等） | 10 个字段直接写在 `qxx_md_item`，不拆子表 | 字段数少且稳定，行业专用系统不需 EAV |
| `item_or_product` 存放位置 | 仅存 `qxx_md_item_type`，物料通过 `item_type_id` 继承 | 消除双写不一致风险 |
| 分类表 `ancestors` 字段 | 删除，仅保留 `parent_type_id` | 圣享最多 2 层分类，MySQL 8.0 递归 CTE 足够 |
| SPU/SKU 实现方式 | **不建独立 SKU 表**，用 `qxx_md_item.parent_id` 自引用实现 | 参考 Qcadoo MES 设计，变体就是另一个物料，库存/BOM/工单零额外逻辑 |
| 标准 BOM 冗余字段 | `qxx_md_product_bom` 不存 `item_code`/`item_name`/`item_spec`，通过 `bom_item_id` JOIN 获取 | 标准 BOM 应实时反映物料主数据 |
| 工单 BOM 冗余字段 | `qxx_pro_workorder_bom` 允许多余字段做快照 | 工单 BOM 独立于标准 BOM，物料改名后工单仍可追溯 |
| 标准 BOM vs 工单 BOM | 单向复制：创建工单时从标准 BOM 复制 → 工单 BOM 可独立修改，不回流 | 客户特殊需求不会污染标准配方 |
| 工序参数 | 四层 EAV：`param_template`(L1定义) → `route_process_param`(L2产品标准，挂 `route_product_id`) → `workorder_param`(L3工单调整) → `feedback_param`(L4实际值) | L2 挂在产品-路线关联上，不同产品即使共享路线也可不同参数值；与 BOM 同模式，单向复制不回流 |
| 路线本身是否需要参数模版 | ❌ 不需要 | 路线字段（编码/描述/工序顺序）固定通用，硬编码列足够；参数模版只在路线工序级 |
| 批次属性配置 `item_id` vs `item_type_id` | ⚠️ 待客户确认：当前挂在 `item_id`（灵活但配置量大），备选方案挂在 `item_type_id`（简洁但不够灵活） | 见 `mes-md.sql` 中 `qxx_md_item_batch_config` 注释 |
| 销售订单表 | ⚠️ 待补：工单的 `source_code` 引用了销售订单号，但 `qxx_sal_order` 表尚未创建 | 客户需求应先落销售订单，再转工单 |
| 物料追溯（Traceability） | `qxx_pro_material_trace` 物料谱系表 + `roll_detail.parent_roll_id` 分切卷自引用 + `card_process.issue_detail_id` 领料直达 | 正向追溯（原料→成品）和反向追溯（成品→原料）双链路覆盖，支持 MySQL 8.0 递归 CTE 查询完整谱系 |

## 纸张复卷明细（Roll Detail）

物料主数据 = 标准规格（`paper_width`/`paper_weight` 存名义值），复卷明细 = 到货实际值（`actual_width`/`actual_weight` 存每卷实测）。两者不冲突：一个是"应该是什么"，一个是"实际是什么"。复卷明细建议挂在入库行或独立 `qxx_wm_roll_detail` 表，不在 `qxx_md_item` 上扩展。

## 工序参数四层体系

每个工序的参数完全不同（印刷=色数/速度/油墨，分切=刀数/宽度/长度），不能硬编码为列。采用 EAV（行即参数）+ 四层单向复制，与 BOM 三层设计同一模式。

```
param_template(L1)  →  route_process_param(L2)  →  workorder_param(L3)  →  feedback_param(L4)
  工序参数定义            产品级标准值                 工单级调整值              实际生产值
  (工艺工程师一次建)       (按产品配置)                (按工单可改)             (操作工每次报工填)
```

| 层 | 表 | 维护者 | 特点 |
|---|---|---|---|
| L1 | `qxx_pro_param_template` | 工艺工程师 | 定义参数schema：编码/名称/类型/单位/枚举值/min/max |
| L2 | `qxx_pro_route_process_param` | 工艺工程师 | 挂在 `route_product_id`（产品-路线关联），不同产品即使共享路线也可不同参数值 |
| L3 | `qxx_pro_workorder_param` | 计划员 | 建工单时从L2复制，`standard_value`(快照)+`adjusted_value`(调整)双字段，可改不回流 |
| L4 | `qxx_pro_feedback_param` | 操作工 | 报工时填实际值，vs L3对比偏差，超min/max→`is_deviation='Y'` |

**Why 四层**: 工单级(L3)独立存在是因为客户定制/返单可能微调工艺参数（如客户要求降印刷速度），调整不应污染产品级标准(L2)，逻辑与 `workorder_bom` 可从 `route_product_bom` 复制后独立修改完全一致。

**路线本身不需要参数模版**: 路线字段（编码/名称/工序顺序/前后关系）固定通用，硬编码列足够。

## 物料追溯体系

**原则**：一张谱系表 + 两个快捷字段，覆盖正向追溯（原料→成品）和反向追溯（成品→原料）。

### 架构总览

```
                    ┌──────────────────────────────────┐
                    │   qxx_pro_material_trace         │
                    │   ┌──────────┐    ┌──────────┐   │
                    │   │ parent   │ →  │ child    │   │
                    │   │ (来源)    │    │ (去向)    │   │
                    │   └──────────┘    └──────────┘   │
                    │   trace_type:                     │
                    │     ISSUE / PRODUCE / SLIT / MERGE│
                    └──────────────────────────────────┘
                           ↑                    ↑
            ┌──────────────┴──┐          ┌──────┴─────────────┐
            │ card_process.    │          │ roll_detail.       │
            │ issue_detail_id  │          │ parent_roll_id     │
            │ (快捷通道)        │          │ (分切自引用)        │
            └─────────────────┘          └────────────────────┘
```

### 三层追溯能力

| 层 | 机制 | 适用场景 |
|----|------|---------|
| **快捷通道** | `card_process.issue_detail_id` → `issue_detail` → `batch` | 单步查询"这个工序用了哪批料"，生产现场高频查询 |
| **谱系表** | `material_trace` 行即边，MySQL 8.0 递归 CTE | 完整追溯图：从原料卷号一直追到客户签收 |
| **自引用** | `roll_detail.parent_roll_id` | 分切场景：一卷大纸切成 N 卷小纸后的父子关系 |

### 正向追溯（Forward Trace）

```
供应商 → 采购单 → 到货入库(qxx_wm_item_recpt)
                         ↓
                   纸卷明细(qxx_wm_roll_detail)  ← 一卷一行，实际规格
                         │
                   批次(qxx_wm_batch)            ← 入库时/排产时生成
                         ↓
    [material_trace: ISSUE] 原料批次 → 流转卡
                         ↓
                   流转卡(qxx_pro_card)          ← 带着 batch_code 走工序
                   流转卡工序(qxx_pro_card_process)  ← 每道工序投入/产出
                         │
                   issue_detail_id → 领料明细    ← 快捷追溯这道工序用了哪批料
                         ↓
    [material_trace: PRODUCE] 流转卡 → 成品批次
                         ↓
                   成品入库(qxx_wm_product_recpt)
                         ↓
                   销售出库(qxx_wm_product_sales) → 客户
```

### 反向追溯（Backward Trace）

```
客户投诉（成品批次 B-001 色差超标）
  → qxx_wm_product_sales (找成品来源工单)
  → qxx_wm_product_recpt (找入库记录)
  → qxx_pro_workorder (找工单)
  → qxx_pro_card + qxx_pro_material_trace (ISSUE) (找消耗的原料批次)
  → qxx_wm_batch (找原料批次)
  → qxx_wm_item_recpt (找入库单)
  → qxx_md_vendor (找供应商)
  → qxx_qc_iqc (找来料检验结果)

快捷版（单表查询）：
  → card_process.issue_detail_id → issue_detail(batch_code) → 锁定原料批次
```

### 分切追溯

```
母卷 RX-001 (门幅1800mm, 长6000m)
  │ parent_roll_id = NULL
  │
  ├── 子卷 RX-002 (切出800mm, 长2000m)
  │     parent_roll_id = RX-001.roll_id
  │
  ├── 子卷 RX-003 (切出800mm, 长2000m)
  │     parent_roll_id = RX-001.roll_id
  │
  └── 子卷 RX-004 (切边余料200mm, 长2000m)
        parent_roll_id = RX-001.roll_id

查询：SELECT * FROM qxx_wm_roll_detail WHERE parent_roll_id = ?;
```

### 与 BOM/参数的复用模式一致

| 设计模式 | BOM | 工序参数 | 物料追溯 |
|---------|-----|---------|---------|
| 单向复制 | 标准BOM→工单BOM | L2→L3 | ISSUE/PRODUCE trace |
| 独立修改不回流 | ✅ | ✅ | ✅ trace 追加不修改 |
| 快照锁定 | 工单BOM冗余字段 | L3 standard_value | trace 记录当时数量 |
